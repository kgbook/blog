前面几篇文章一直在讲 Harness Engineering 如何提高效率：写规则、跑检查、建反馈、做循环、管熵。

但如果只讲好处，会给人一种错觉：只要 Harness 足够好，AI Agent 就能可靠替代工程师完成所有工作。

这不是真的。

Harness Engineering 很有价值，但它有边界。理解边界，和掌握方法一样重要。否则你会在错误场景里过度自动化，把效率工具变成风险放大器。

这篇文章讲三个最重要的边界：

1. 行为正确性很难验证。
2. Harness 会和模型能力耦合。
3. 人类不会消失，但角色会改变。

## 测试通过不等于正确

AI 编程里最危险的一句话是：

```text
All tests pass.
```

这句话当然是好消息，但它不等于功能真的正确。

测试只能验证你写出来的场景。它不能保证：

- 需求理解没有偏差。
- 用户真正需要这个功能。
- 抽象方向是对的。
- 安全边界是合理的。
- 长期维护成本可接受。

举个简单例子。

你让 AI 做一个“导出账单”功能，测试包括：

- 有数据时导出 CSV。
- 没数据时导出空文件。
- 金额格式正确。

测试都通过了。但用户真正需要的是“按当前筛选条件导出”，AI 却导出了全部数据。测试没覆盖这个需求，所以它不会失败。

这不是测试没用，而是说明测试无法替代需求判断。

## 三层验证

可以把 Harness 的验证能力分成三层。

### 第一层：可维护性验证

这一层最成熟。

包括：

- 格式化
- lint
- 类型检查
- 文件大小
- 命名规则
- 重复代码扫描

这些问题判断标准清楚，适合机器检查。

### 第二层：架构适应度验证

这一层中等成熟。

包括：

- 模块依赖方向
- 分层边界
- 禁止循环依赖
- API 兼容性
- 数据访问路径

这些也能部分自动化，但需要你先定义架构规则。

### 第三层：行为正确性验证

这一层最难。

包括：

- 功能是否解决真实问题。
- 用户体验是否合理。
- 产品语义是否正确。
- 业务边界是否完整。
- 安全和合规风险是否处理。

这层很难完全机械化。AI 可以辅助审查，但人类仍然要负责最终判断。

## 为什么行为验证这么难

因为行为正确性通常依赖未完全写下来的世界知识。

例如：

```text
“用户删除项目时，应该怎么处理历史报表？”
```

这个问题不是代码问题，而是产品和业务问题。不同产品可能有不同答案：

- 级联删除。
- 禁止删除。
- 软删除。
- 保留报表快照。
- 需要管理员审批。

AI 可以给建议，但不能替你决定。

如果你没有把规则写进任务或文档，AI 只能根据通用经验猜。猜出来的代码也许测试通过，但产品语义错了。

这就是 Harness Engineering 的最大边界：它擅长约束已知规则，不擅长替你发现所有未知规则。

## LLM-as-judge 不是银弹

一种看起来诱人的方案是：让另一个 LLM 当评委。

例如：

```text
请审查这个实现是否满足需求。
```

这确实有用。独立 Reviewer Agent 经常能发现遗漏测试、边界情况、架构问题。

但它不是银弹：

- 它也可能误解需求。
- 它可能过度称赞。
- 它可能偏好表面完整的答案。
- 它需要好的评估标准。

评估者也是模型，仍然需要 Harness。

更可靠的组合是：

```text
确定性检查 + Reviewer Agent + 人类验收
```

不要把最后一层判断完全交给模型。

## Model-Harness 耦合

第二个边界是：Harness 会和模型能力耦合。

某个 Harness 设计，往往隐含了对模型的判断：

```text
模型容易忘记上下文，所以加 scratchpad。
模型不会自我审查，所以加 Reviewer。
模型不会规划长任务，所以加 Planner。
模型容易提前收尾，所以加完成门控。
```

当模型升级后，这些假设可能变化。

旧模型需要很强的 Planner，新模型可能自己规划得很好。旧模型需要频繁 context reset，新模型可能不需要。某个复杂流程在旧模型上提升质量，在新模型上反而拖慢速度。

所以 Harness 不是一次设计，永久有效。它有生命周期。

## Harness 也需要重构

我们常说代码要重构，但 Harness 本身也要重构。

需要定期检查：

- 哪些规则已经没必要？
- 哪些检查误报太多？
- 哪些提示词过时？
- 哪些流程只是补偿旧模型缺陷？
- 哪些工具成本太高但收益很低？

如果不清理，Harness 会变成负担。

一个简单做法是每次模型或工具升级后，跑几类基准任务：

```text
小功能实现
bug 修复
重构
测试补齐
文档生成
```

观察旧 Harness 是否仍然有帮助。如果某个步骤不再增加质量，就考虑删掉或降级。

## 约束越强，不一定越好

Harness Engineering 很强调约束，但约束不是越多越好。

太少的问题是 AI 乱跑。

太多的问题是 AI 被绑死：

- 上下文被规则挤满。
- 简单任务也要跑复杂流程。
- Agent 不敢利用新能力。
- 维护成本越来越高。

好的 Harness 应该像护栏，不像迷宫。

它约束结果和边界，而不是规定每一步动作。

## 人类的位置变了

Harness Engineering 不是让人退出软件开发，而是改变人的位置。

过去：

```text
人类写大部分代码。
工具负责执行和检查。
```

现在：

```text
人类设计目标、边界和验收。
AI 生成实现。
工具执行检查。
人类判断方向和质量。
```

人的价值从“敲代码”更多转向：

- 定义问题。
- 拆解任务。
- 设计架构。
- 建立规则。
- 判断取舍。
- 识别坏味道。
- 验收用户价值。

这不是降级，而是上移。

## 什么任务适合交给 Agent

可以用一个矩阵判断。

| 任务类型 | 需求清晰 | 可验证 | 适合程度 |
|---|---|---|---|
| CRUD 接口 | 高 | 高 | 很适合 |
| 测试补齐 | 高 | 高 | 很适合 |
| 小型重构 | 中高 | 高 | 适合 |
| UI 微调 | 中 | 中 | 谨慎 |
| 产品方案 | 低 | 低 | 不适合全自动 |
| 安全审计 | 中 | 低 | 只能辅助 |
| 核心架构 | 低中 | 低中 | 人类主导 |

判断口诀：

```text
目标越清楚、验证越强，越适合 Agent。
目标越模糊、验证越弱，越需要人类主导。
```

## 失败案例通常长什么样

Harness 不足时，失败往往不是“代码完全不能跑”，而是更隐蔽：

- 功能表面完成，但需求理解错。
- 测试全绿，但测试覆盖的是错误需求。
- 架构能跑，但未来难改。
- 文档写得漂亮，但和代码不一致。
- Agent 自评很好，但人类一看不对。

所以不要只看“能不能跑”，还要看“是不是该这么做”。

## 成熟使用者的心态

成熟使用 Harness Engineering，不是盲目相信 AI，也不是排斥 AI。

而是把 AI 看成一个高吞吐执行者：

- 给它明确任务。
- 给它足够上下文。
- 给它工具。
- 给它检查。
- 让它反馈失败。
- 由你决定方向。

这和管理一个初级但速度极快的团队成员有点像。你不会让他决定公司战略，但会让他在清晰边界里完成大量工作。

## 未来趋势

我认为 Harness Engineering 会继续往三个方向发展。

### 1. Harness 产品化

更多工具会内置：

- 项目规则读取
- 自动测试修复
- 多 Agent 审查
- 任务状态持久化
- 沙箱执行
- 权限控制

也就是说，很多手写 Harness 会变成产品功能。

### 2. Meta-Harness

未来可能不是人手写每个 Harness，而是系统根据任务自动组合：

```text
这个任务需要 Planner。
这个任务需要 Browser。
这个任务需要 Reviewer。
这个任务只需要快速修复。
```

Harness 本身会被动态生成和调度。

### 3. 人类更专注于评估标准

当实现越来越自动化，人类最重要的产出会变成：

- 什么是好结果？
- 什么不能破坏？
- 哪些风险不可接受？
- 如何验证用户价值？

谁能更清楚地定义问题，谁就能更好地驾驭 AI。

## 小结

Harness Engineering 很强，但不是魔法。

它能显著改善：

- 上下文管理
- 规则执行
- 测试反馈
- 重复任务
- 代码熵管理
- Agent 编排

但它不能自动解决：

- 模糊需求
- 产品判断
- 行为正确性
- 价值取舍
- 长期架构方向

最终的原则是：

```text
让机器检查确定的事。
让 AI 执行清楚的事。
让人类判断重要的事。
```

这才是 Harness Engineering 的成熟形态。
